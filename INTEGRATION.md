# אינטגרציה: עדכון משחק פעיל ממערכת יצירת המשחקים ("פוש רענון")

מסמך זה מגדיר מה מערכת יצירת המשחקים צריכה לעשות כדי שעדכון בקובץ ה-JSON של
משחק **ישתקף מיידית במשחק שרץ ברגע זה** — שאלות, מדיה, הגדרות, שמות וקבוצות —
בלי לאבד ניקוד, הצבעות או מיקום.

## העיקרון

עמוד המשחק הוא דפדפן — הוא לא יכול לקבל HTTP נכנס. לכן הוא **מתחבר החוצה**
לערוץ עדכונים שמערכת היצירה מספקת, ומחכה לאות. כשמגיע אות — המשחק מושך מחדש
את ה-JSON מאותה כתובת `?game=` (עם `cache: no-store`) ומחיל אותו "חם" דרך
`engine.updateGame` (בלי remount).

## שלב 1 — בניית כתובת המשחק

בעת פתיחת/הפצת משחק, מוסיפים לכתובת את פרמטר `push`:

```
https://gamemwemaster.caprover.clicker.co.il/?game=<כתובת-ה-JSON>&push=<כתובת-ערוץ-העדכונים>
```

- `push` שמתחיל ב-`wss://` / `ws://` → המשחק נפתח ב-**WebSocket**.
- כל כתובת אחרת (`https://...`) → המשחק נפתח ב-**SSE** (EventSource). זו
  האפשרות הפשוטה ביותר.
- מומלץ ערוץ פר-משחק, למשל: `https://your-server/push/<gameId>`.

## שלב 2 — מה לעשות בכל שמירה במערכת היצירה

1. לעדכן את קובץ ה-JSON **באותה כתובת** שהמשחק טוען ממנה.
2. לשדר הודעה בערוץ של אותו `gameId`. תוכן ההודעה — הכי פשוט:

   ```
   data: update

   ```

   כל אחת מהמילים `update` / `refresh` / `reload` / `refetch` / `ping` — או
   אפילו הודעה ריקה — נחשבת אות רענון, והמשחק מושך מחדש את הקובץ.

אפשרות מתקדמת: לשלוח את ה-JSON המלא בהודעה עצמה — `{"game": { ...קובץ המשחק... }}`
— ואז המשחק מחיל אותו ישירות בלי משיכה חוזרת. **לא מומלץ**: משיכה-חוזרת שומרת
מקור אמת אחד (הקובץ) והודעות הערוץ נשארות זעירות.

## שלב 3 — שרת ה-relay (מינימלי)

נדרש endpoint אחד של SSE (המשחק מתחבר אליו) + endpoint פנימי לשידור (מערכת
היצירה קוראת לו אחרי כל שמירה). דוגמת עיקרון ב-Node/Express:

```js
const subs = new Map(); // gameId -> Set<res>

// המשחק מתחבר לכאן (SSE)
app.get('/push/:gameId', (req, res) => {
  res.writeHead(200, {
    'Content-Type': 'text/event-stream',
    'Cache-Control': 'no-cache',
    'Access-Control-Allow-Origin': '*', // חובה — המשחק רץ בדומיין אחר
  });
  res.write('retry: 3000\n\n');
  if (!subs.has(req.params.gameId)) subs.set(req.params.gameId, new Set());
  subs.get(req.params.gameId).add(res);
  req.on('close', () => subs.get(req.params.gameId)?.delete(res));
});

// מערכת היצירה קוראת לכאן אחרי כל שמירה
app.post('/notify/:gameId', (req, res) => {
  for (const s of subs.get(req.params.gameId) ?? []) s.write('data: update\n\n');
  res.sendStatus(204);
});
```

זרימה: שמירה במערכת היצירה → עדכון הקובץ → `POST /notify/<gameId>` → כל משחק
פעיל של אותו gameId מתרענן תוך שנייה. אם אין משחק פעיל — אין מנויים וה-POST
לא עושה דבר. ניתוקים אינם בעיה: EventSource מתחבר מחדש לבד, וב-WebSocket יש
בצד המשחק התחברות-חוזרת עם השהיה גוברת.

## חלופה בלי שרת relay — postMessage

אם מערכת היצירה פותחת את המשחק בעצמה (`window.open` או iframe), אין צורך
ב-`&push=` כלל:

```js
gameWindow.postMessage({ type: 'trivia-refresh' }, '*');
// או עם הקובץ עצמו (החלה ישירה בלי משיכה חוזרת):
gameWindow.postMessage({ type: 'trivia-refresh', game: updatedGameJson }, '*');
```

הודעות בלי המעטפת `type: 'trivia-refresh'` — המשחק מתעלם מהן.

## כללים שחובה לשמור בעדכון משחק פעיל

1. **אותו `id` משחק** — לעולם לא לשנות אותו בעדכון (הגיבוי והמרשם ממופתחים
   לפיו).
2. **מזהי שקופיות (`id`) יציבים** — עריכה/הוספה/מחיקה/סידור-מחדש בטוחים;
   **מספור-מחדש אסור** (יגרום לקפיצת מיקום — המיקום נשמר לפי מזהה השקופית).
3. **שדה `users` הוא מחרוזת JSON** (לא אובייקט):
   ```json
   "users": "{\"0501234567\":{\"name\":\"דנה\",\"groupName\":\"אדומים\"}}"
   ```
   המיזוג הוא הוספה/עדכון (upsert לפי מספר; קטגוריה/קבוצות לפי שם) — מחיקת
   שחקן מה-JSON **לא** מוחקת אותו מהמרשם של משחק שכבר רץ.
4. **Cache** — להגיש את ה-JSON עם `Cache-Control: no-store` (או לוודא שה-CDN
   מכבד revalidation), כדי שהמשיכה-החוזרת תקבל את הגרסה הטרייה.
5. **CORS** — גם כתובת ה-JSON וגם ערוץ ה-push חייבים
   `Access-Control-Allow-Origin` שמתיר את דומיין המשחק.
6. עדיף להימנע ממחיקת תשובה בשאלה שההצבעה עליה פתוחה ממש עכשיו.

## מה מתעדכן ומה לא (במשחק שכבר רץ)

| תחום | מתעדכן? | הערות |
|------|---------|-------|
| שאלות (עריכה/הוספה/מחיקה/סידור) | ✅ מיידי | המיקום נשמר לפי מזהה שקופית; שקופית נוכחית שנמחקה → נפילה נקייה לקרובה |
| מדיה (פתיחה/סיום/רקעים/לוגו) | ✅ מיידי | נטענת מראש ברקע |
| הגדרות גלובליות (צבעים, כותרת, סאונדים…) | ✅ מיידי | |
| שמות וקבוצות (`users`) | ✅ מיידי | מיזוג למרשם; עריכות ידניות של המפעיל נשמרות |
| ניקוד / הצבעות / מיקום / שלבי חשיפה | 🛡 נשמרים | לא מושפעים מהרענון |
| טיימר רץ | 🛡 מוגן | רענון באמצע הצבעה לא מאפס את הטיימר |
| `autoTransition` | ❌ | באמצע משחק זו הגדרת המפעיל (עם דריסה שמורה per-game) |
| קוד חדר (`room`) | ❌ | הסוקט לא מתחבר מחדש באמצע משחק |

## מימוש בצד המשחק (למתחזקים)

- `src/app/pushChannel.ts` — פתיחת הערוץ (postMessage / SSE / WS) ופענוח
  ההודעות (`interpretPushMessage`, נבדק ביחידה).
- `src/app/App.tsx` — `applyGame`: מיזוג `users` למרשם + `setGame`;
  `refetchGame`: משיכה חוזרת מ-`?game=` עם `no-store`.
- `src/app/GameHost.tsx` — effect הרענון החם: `engine.updateGame` + טעינת
  המרשם המעודכן.
- `src/engine/gameEngine.ts` — `updateGame`: החלפת תוכן עם שימור מלא של
  ניקוד/הצבעות/מיקום.
